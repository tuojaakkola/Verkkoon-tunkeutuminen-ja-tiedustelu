## a) Apache-palvelimen asennus ja lokit

Aloitin tehtävän asentamalla apachen komennolla:
`sudo apt install apache2`

Lataus onnistui ja navigoin apachen oletussivulle joka löytyy osoitteesta `http://localhost/`

En aluksi tiennyt miten voin tutkia apachen tietoja. 
Apache lataa lokitiedostot kansioon `/var/log/apache2/access.log` joten siirryin sinne. Tulostin log rivejä tail komenolla jolloin sivulle terminaaliin tuli useita lokirivejä:
`sudo tail -n 20 apache2/access.log`



**Lokirivistä näkee:**
* **127.0.0.1** = Asiakkaan IP-osoite. Pyyntö tuli omalta koneeltani eli localhostista.
* **RFC 1413 identiteetti** = Tietoa ei saatavilla.
* **Käyttäjätunnus** = Sivusto ei vaadi kirjautumista.
* **[02/Apr/2026:10:17:45 +0000] Aikaleima** = Lataus tapahtui 2. huhtikuuta 2026 klo 10:17:45 UTC-aikaa.
* **"GET / HTTP/1.1" Pyyntö** = Selain käytti tiedon hakuun GET-metodia. `/` tarkoittaa, että haettiin palvelimen juurta. Käytetty protokolla oli HTTP/1.1.
* **200 HTTP-tilakoodi** = Kaikki meni hyvin ja palvelin toimitti pyydetyn sivun.
* **3383 Koko** = Palvelimen lähettämän sivun koko oli 3383 tavua.
* **"-" (Referer)** = Sivuille ei tultu toisen sivuston linkkiä painamalla, vaan osoite kirjoitettiin suoraan selaimeen ja siksi tyhjä.
* **"Mozilla/5.0 (X11; Linux x86_64; rv:140.0)..." User-Agent** = Selain lähetti tiedot itsestään.

## b) Nmap-porttiskannaus

Aloitin tehtävän tutustumalla mikä on nmap. Latasin sen komennolla:
`sudo apt install nmap`

Sitten skannasin sillä porttia 80 komennolla `nmap -A -p 80 localhost` niinkuin tehtävänannossa pyydettiin.
* lippu `-A` = Agressive
* lippu `-p 80` = Port 80



**Nmap-skannauksen tulokset:**
* **PORT 80/tcp state on open** eli skannattu portti on auki ja kuuntelee. Tämä todistaa, että weppipalvelin on päällä ja toimii.
* **SERVICE http:** Nmap tunnisti, että portissa pyörii weppipalvelu.
* **VERSION Apache httpd 2.4.66 ((Debian)):** Nmap sai selville tarkan palvelinohjelmiston (Apache) version (2.4.66) ja käyttöjärjestelmän (Debian).

Nmap totesi portin olevan auki ja selvitti -A -parametrin avulla tarkan palvelinohjelmiston ja käyttöjärjestelmän.

## c) Nmap-skriptit (NSE)

Kun käytin `-A` lippua niin Nmap ajoi automaattisesti seuraavat kaksi skriptiä:
* **http-server-header:** Skripti nappasi talteen palvelimen antaman tunnistetiedon (Apache/2.4.66 (Debian)).
* **http-title:** Skripti kävi lukemassa verkkosivun otsikon ("Apache2 Debian Default Page: It works").

## d) Jäljet lokeissa (Grep)

Käytin lokien tutkimiseen komentoa:
`sudo grep -i nmap /var/log/apache2/access.log` 

Lippu `-i` = ignore case eli ei välitä isostoista kirjaimista vaan etsii kaikki. Komento löysi useita nmap osumia.

Nmap teki samalla sekunnilla kymmeniä pyyntöjä ja yritti automaattisesti etsiä palvelimelta piilotettuja kansioita ja tunnettuja haavoittuvuuksia kuten `GET /.git/HEAD` ja `GET /robots.txt`. Suurin osa näistä tuotti virhekoodin 404.   
Tavallinen käyttäjä ei klikkaile kymmeniä linkkejä sekunnissa joten tälläinen nuuhkinta voitaisiin estää **rate limiting** avulla.

## e) Verkkoliikenteen analyysi (Wireshark)

Etsin Wiresharkista skannauksen jälkiä nauhoittamalla **loopback** verkkokorttia ja ajamalla porttiskannauksen uudestaan. Etsin nmap nimisiä paketteja wiresharkin filttereiden avulla ja löysin yhteensä 25 HTTP-pakettia.

**Tutkimastani paketista paljastui kaksi selvää Nmapin jälkeä:**
* **Keksitty tiedostonimi:** Nmap yritti hakea olematonta tiedostoa nimeltä `/nmaplowercheck...`. Tämä ilmeisesti testaa, miten palvelin reagoi virheisiin. 
* **Skanneri:** Paketin tiedoissa (User-Agent -kentässä) lukee avoimesti "Nmap Scripting Engine". Skanneri siis kertoo palvelimelle oman nimensä.

## f) Reaaliaikainen seuranta (ngrep)

Aloitin lataamalla ngrep komennolla `sudo apt install ngrep`. Sitten avasin kaksi terminaalia. Ensimmäisessä laitoin ngrep-työkalun kuuntelemaan paikallista verkkoliikennettä komennolla: 
`sudo ngrep -W byline -d lo -i "nmap" port 80` 

Parametri `-W byline` tekee tulosteesta ihmisluettavaa, `-d lo` valitsee oikean verkkokortin ja `-i "nmap"` suodattaa esiin vain ne paketit, jotka sisältävät kyseisen sanan. Tämän jälkeen ajoin toisessa terminaalissa Nmap-skannauksen (`nmap -A -p 80 localhost`).

Heti skannauksen alettua ngrep tulosti ruudulle reaaliajassa Nmapin lähettämiä HTTP-pyyntöjä. Tulosteesta näkee samat asiat kuin aiemmin Wiresharkissa ja lokeissa eli User-Agent paljastaa taas nmapin.

## g) Jälkien peittäminen (User-Agent Spoofing)

Käytin nmapin user agentin vaihtamiseen `--script-args` parametriä jonka avulla voi piilottaa "Nmap Scripting Engine" tunnisteen. Lisäsin parametreiksi:
`http.useragent="Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/120.0.0.0 Safari/537.36" localhost`

Kun ajoin skannauksen ja tarkistin sen jälkeen Apachen `access.log` -tiedoston, huomasin huijauksen onnistuneen. Lokiriveiltä ei enää löytynyt sanaa "Nmap". Sen sijaan jokaisen pyynnön perässä (User-Agent -kentässä) oli äsken syöttämäni Chrome-selaimen pitkä merkkijono mikä nykyaikaisissa selaimissa tulee.
